# PR strategies — per-strategy behaviour

One of three strategies elected at Preflight step 7: `from-start`, `at-end` *(default)*, or `none`. The election affects:

1. When a PR is created.
2. What happens at final completion.

There is no end-of-sprint behaviour because there are no sprints. The parcel runs as one uninterrupted unit; the only PR action between Preflight and completion is the optional `from-start` opening.

The election is fixed for the parcel. Changing it mid-execution is a plan-scope change — offer `/plan`; never rewrite it in place.

## Strategy summary

| Strategy | PR created | Final completion |
|---|---|---|
| `at-end` *(default)* | Final wrap-up only | Open the PR with consolidated test plan |
| `from-start` | Immediately after Preflight | Strip `[In Flight]`, remove `in-flight` / `do-not-merge` labels |
| `none` | Never (user handles it) | Tell the user the branch is ready |

## Creating the from-start PR

Runs once, immediately after Preflight, before the first code commit. Unlike the sprint-loop variant, this fires right after Preflight — there is no Sprint Refinement step to delay it.

### Step 1 — Check for an existing PR

```
gh pr view --json url,number,title,labels
```

One already exists for this branch? Reuse — record the URL in the tracker header, skip creation. Never open a second PR for a branch; it splits the review channel.

### Step 2 — Create the PR

Non-draft. Draft PRs don't trigger the review and notification channels that make `from-start` valuable — the point is live visibility from commit zero.

```
gh pr create \
  --title "[In Flight] <plan title>" \
  --body "<body>" \
  --label in-flight \
  --label do-not-merge
```

The `in-flight` and `do-not-merge` labels are load-bearing: they're how reviewers know this PR is mid-work and not yet mergeable despite not being a draft. Both were created at Preflight step 10.

### Step 3 — Body template

Three sections:

- `## Goal` — copied from the plan's Overview.
- `## Phase Status` — one checklist item per plan phase, all unchecked. The agent ticks each as that phase's Automated Success Criteria pass.
- `## Manual Test Plan` — empty placeholder; populated at end-of-parcel review when Manual SC are run with the user.

No automation-authored banner (no "generated by X", no tool signature). The PR is team-neutral — reviewers see the work, not a label saying who wrote it.

### Step 4 — Record in the tracker

Record the PR URL on the header line `**PR:** <URL>` (was `none yet`).

## Manual test plan format

`from-start` populates the Manual Test Plan at end-of-parcel review; `at-end` populates it on PR creation; `none` skips it. Same format regardless:

A numbered step-by-step checklist covering the golden path and important edge cases, drawn from each phase's Manual Success Criteria, in plan-defined phase order. One subsection per phase: `### Phase N — <name>`.

Every main-list item uses a markdown checkbox (`1. [ ] …`) so the reviewer can tick items off. GitHub renders `1. [ ] …` correctly; a bare numbered list without `[ ]` isn't a usable checklist.

Every main-list item must be verifiable against the state at the moment the PR opens. Items that need post-release or post-merge setup that hasn't happened yet don't belong in the main list — put them under `### Not Locally Testable` with a clear reason ("requires a GitHub Release to be drafted first"). A plan that asks the reviewer to do manual setup just to make an item testable isn't a test plan.

## Final-completion behaviour

Fires when every phase's Automated Success Criteria pass and the user has confirmed every Manual Success Criterion at the end-of-parcel review, after the tracker's Status is set to `Complete`.

### at-end *(default)*

Open the PR now:

```
gh pr create \
  --title "<plan title>" \
  --body "<body>" \
  --base main
```

Body consolidates every phase's manual test plan into one end-to-end plan, structured as `### Phase N — <name>` subsections. Don't open silently; offer first ("Ready to open the PR with this body?"). The strategy's name is `at-end` and the offer respects that.

### from-start

PR already exists (opened after Preflight). Edit it:

- Strip the `[In Flight]` prefix from the title.
- Remove the `in-flight` and `do-not-merge` labels.

```
gh pr edit --title "<title without prefix>" --remove-label in-flight --remove-label do-not-merge
```

Body already contains the consolidated test plan filled in at end-of-parcel review.

### none

Tell the user the branch is ready and the commits are on the remote. Handoff complete.
