# PR strategies — per-strategy behaviour

One of four strategies is elected during Preflight step 7: `from-start`, `at-end`, `per-sprint`, or `none`. The election changes three things during the session:

1. **When a PR is created.**
2. **What happens at the end of each sprint.** (See `references/feedback-loop.md` and the Execute section of SKILL.md.)
3. **What happens at final completion.** (Same.)

This file covers step 1. Steps 2 and 3 live alongside the sprint loop because they fire per-sprint, not at creation time.

## When PRs are created

| Strategy | Created | Title style |
|---|---|---|
| `from-start` | After Sprint Refinement, before first code commit | `[In Flight] <plan title>` until wrap-up |
| `per-sprint` | End of each sprint | Sprint name, no prefix |
| `at-end` | Final wrap-up only | Plan title |
| `none` | Never (user handles it) | n/a |

The election is fixed for the session. Changing it mid-implementation is a plan-scope change — offer `/plan`; never rewrite it in place.

## Creating the from-start PR

Runs once, immediately after Sprint Refinement, before the first code commit. The body needs the refined sprint list, which is why this is not a Preflight step.

### Step 1 — Check for an existing PR

```
gh pr view --json url,number,title,labels
```

If one already exists for this branch, reuse it — record the URL in the tracker header and in `Preflight Decisions`, then skip creation. Never open a second PR for the same branch; it splits the review channel and confuses reviewers.

### Step 2 — Create the PR

Non-draft. Draft PRs don't trigger the review and notification channels that make `from-start` valuable in the first place — the point is live visibility from commit zero.

```
gh pr create \
  --title "[In Flight] <plan title>" \
  --body "<body>" \
  --label in-flight \
  --label do-not-merge
```

The `in-flight` and `do-not-merge` labels are load-bearing: they are how reviewers know this PR is mid-work and not yet mergeable, even though it is not a draft. Both labels were created during Preflight step 10.

### Step 3 — Body template

The body has three sections:

- `## Goal` — copied from the plan's Overview.
- `## Sprints` — one checklist item per refined sprint, all unchecked.
- `## Manual Test Plan` — empty placeholder. Populated as sprints complete; consolidated at wrap-up.

Do not include an automation-authored banner (no "generated by X", no tool signature). The PR is team-neutral — reviewers should see work, not a label telling them who wrote it.

### Step 4 — Record in the tracker

After creation, record the PR URL in two places in the tracker:

- The header line `**PR:** <URL>` (was `none yet`).
- `Preflight Decisions → PR` (if that slot is present in your tracker variant; otherwise the header is sufficient).
