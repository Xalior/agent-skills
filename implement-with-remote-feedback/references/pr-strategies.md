# PR strategies — per-strategy behaviour

One of four strategies is elected during Preflight step 7: `from-start`, `at-end`, `per-sprint`, or `none`. The election affects three things:

1. **When a PR is created.**
2. **What happens at the end of each sprint.**
3. **What happens at final completion.**

The election is fixed for the session. Changing it mid-implementation is a plan-scope change — offer `/plan`; never rewrite it in place.

## Strategy summary

| Strategy | PR created | End-of-sprint | Final completion |
|---|---|---|---|
| `from-start` | After Sprint Refinement, before first code commit | Tick the sprint's checkbox, append its test plan to the PR body | Strip `[In Flight]` prefix, remove `in-flight` / `do-not-merge` labels |
| `per-sprint` | End of each sprint | Open a new standalone PR for that sprint | Confirm all sprint PRs are merged or queued; tell the user |
| `at-end` | Final wrap-up only | No PR action | Offer to open the final PR |
| `none` | Never (user handles it) | No PR action | Tell the user the branch is ready |

## Creating the from-start PR

Runs once, immediately after Sprint Refinement, before the first code commit. The body needs the refined sprint list, which is why this is not a Preflight step.

### Step 1 — Check for an existing PR

```
gh pr view --json url,number,title,labels
```

If one already exists for this branch, reuse it — record the URL in the tracker header, then skip creation. Never open a second PR for the same branch; it splits the review channel and confuses reviewers.

### Step 2 — Create the PR

Non-draft. Draft PRs don't trigger the review and notification channels that make `from-start` valuable in the first place — the point is live visibility from commit zero.

```
gh pr create \
  --title "[In Flight] <plan title>" \
  --body "<body>" \
  --label in-flight \
  --label do-not-merge
```

The `in-flight` and `do-not-merge` labels are load-bearing: they are how reviewers know this PR is mid-work and not yet mergeable, even though it is not a draft. Both labels were created during Preflight step 10.

### Step 3 — Body template

Three sections:

- `## Goal` — copied from the plan's Overview.
- `## Sprints` — one checklist item per refined sprint, all unchecked.
- `## Manual Test Plan` — empty placeholder; populated as sprints complete.

Do not include an automation-authored banner (no "generated by X", no tool signature). The PR is team-neutral — reviewers should see the work, not a label telling them who wrote it.

### Step 4 — Record in the tracker

Record the PR URL on the tracker header line `**PR:** <URL>` (was `none yet`).

## End-of-sprint behaviour

Fires at the end-of-sprint step in SKILL.md → Execute sprints, once Automated Success Criteria pass and the sprint-completion plan audit is done.

### from-start

Edit the existing PR body:

- Tick that sprint's checkbox under `## Sprints`.
- Append the sprint's manual test plan under `## Manual Test Plan`, under a heading `### Sprint N — <name>`.

Do not open a new PR.

### per-sprint

Open a new PR for this sprint's changes:

```
gh pr create --title "<sprint name>" --body "<body>" --base main
```

- No `[In Flight]` prefix — the sprint is complete.
- Body contains: the sprint's name, what it covers, its manual test plan.
- Target `main`.
- If the prior sprint's PR is still unmerged, stop — that is a blocker. A new PR stacked on an unmerged PR makes review ambiguous, since reviewers can't tell which changes are for which sprint.

### at-end / none

No PR action at sprint end. Continue to the next sprint.

## Manual test plan format

All strategies except `none` eventually produce a manual test plan (attached incrementally to the `from-start` PR, to each `per-sprint` PR, or to the final `at-end` PR). The format is the same:

The plan is a numbered, step-by-step checklist covering the golden path and the important edge cases from the sprint's Success Criteria.

Every main-list item uses a markdown checkbox (`1. [ ] …`) so the reviewer can tick items off as they verify. GitHub renders `1. [ ] …` correctly; a bare numbered list without `[ ]` isn't usable as a checklist — a reviewer can't track their own progress on it.

Every main-list item must be verifiable against the state at the moment the PR is opened. Items that require post-release or post-merge setup that hasn't happened yet don't belong in the main list — they go under a `### Not Locally Testable` subsection with a clear reason ("requires a GitHub Release to be drafted first"). A test plan that asks the reviewer to do manual setup just to make an item testable isn't a test plan.

## Final-completion behaviour

Fires when all sprints are complete and all phase-level Success Criteria are cumulatively met, after the tracker's Status is set to `Complete`.

### from-start

Edit the PR: strip the `[In Flight]` prefix from the title; remove the `in-flight` and `do-not-merge` labels.

```
gh pr edit --title "<title without prefix>" --remove-label in-flight --remove-label do-not-merge
```

The body already contains the consolidated test plan from sprint ends — nothing to append.

### per-sprint

Every sprint's PR is already open and standalone. Confirm all are merged or queued for merge; tell the user the state. This is the only strategy where "complete" doesn't mean "one PR ready to merge" — it means N PRs, each independently mergeable.

### at-end

Offer to open the final PR now. Body consolidates each sprint's manual test plan into one end-to-end plan. Do not open the PR silently; the strategy's name is `at-end`, and the offer respects that.

### none

Tell the user the branch is ready and the commits are on the remote. Handoff is complete.
